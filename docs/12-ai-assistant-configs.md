# 12. AI-assistant configs (Claude Code skill + Cursor rules)

The drop-in `kerf.claude-skill.md` and `kerf.cursorrules` from KF-128 told
AI assistants in *downstream* projects how to write kerf code. KF-128
shipped them as repo-root files: a consumer had to find them on GitHub
and copy them by hand. That funnel loses most users — the maintainer's
own `~/Documents/hotsheet` project ran a kerf-authoring audit (HS-8449)
without ever loading the skill, because no copy had been installed — and
once a file *is* copied it never resynchronises across kerfjs version
bumps. The drop-ins go stale silently.

This doc is the canonical statement of how those configs ship now: the
files live inside the published npm package, they carry a structured
version + marker contract so updates are detectable, and an ESLint rule
in `eslint-plugin-kerfjs` surfaces drift on every lint pass with an
auto-fix that preserves consumer customisations.

## 12.1 One feature, three components

The work is intentionally one feature with three pieces, not three
independent improvements:

1. **Bundling** (§12.2). The canonical configs ship inside the
   `kerfjs` npm package so they land on disk via the same
   `npm install kerfjs` that the consumer already runs.
2. **Canonical-file contract** (§12.3). Each bundled file carries a
   `kerf-skill-version` line and a structural marker. The version is
   the staleness signal; the marker is the boundary between "what kerf
   maintains" and "what the consumer has added."
3. **ESLint rule** (§12.4). `kerfjs/ai-assistant-configs` in
   `eslint-plugin-kerfjs` reads the bundled manifest on every lint pass
   and reports missing or stale files; `eslint --fix` replaces only
   the canonical section (above the marker), leaving the consumer's
   append zone untouched.

Any single piece in isolation is weaker:

- Bundling without the rule is just a moved file — no nudge to
  install, no signal when it goes stale.
- The rule without bundling either duplicates the canonical content
  inside the eslint plugin (a second version-drift surface) or has
  to fetch it (offline-hostile).
- The contract without both is unused metadata.

## 12.2 The bundle inside the kerfjs npm package

The published `kerfjs` package contains an `ai/` directory at the
package root:

```
kerfjs/
├── dist/                ← JS bundles, types, source maps
├── ai/
│   ├── skill.md         ← Claude Code skill (copy of kerf.claude-skill.md)
│   ├── cursorrules      ← Cursor rules (copy of kerf.cursorrules)
│   └── manifest.json    ← version + sha256 + dest for each bundled file
└── …
```

`ai/` is at the package root (not under `dist/`) so the import path for
tooling is `kerfjs/ai/manifest.json`, not `kerfjs/dist/ai/…`. Resolved
from a downstream consumer's project root with:

```js
require.resolve('kerfjs/ai/manifest.json', { paths: [cwd] })
```

### 12.2.1 Source of truth at the repo root; mirror under `ai/`

The repo-root `kerf.claude-skill.md` and `kerf.cursorrules` remain the
human-edited source of truth. They stay at the root so GitHub browsers
and the [`llms.txt`](../llms.txt) index keep finding them — a consumer
who arrives on GitHub before installing should not have to descend into
a publish-time directory.

`ai/skill.md` and `ai/cursorrules` are byte-for-byte mirrors of the root
files, regenerated by `scripts/sync-ai-bundle.mjs` and committed to git
alongside the source of truth. The duplication is deliberate: the
mirror IS what consumers receive via npm, and committing it lets
GitHub-browsers see exactly the file that will land in their
`node_modules`. The mirror is kept honest by `npm run
check:ai-bundle-in-sync` (`scripts/check-ai-bundle.mjs`), wired into
`npm run check` next to the existing
[`check:docs:test-inventory`](../scripts/check-doc-test-inventory.mjs) /
[`check:docs:api-coverage`](../scripts/check-doc-api-coverage.mjs)
gates.

If a contributor edits a root file without re-running the sync (or
edits a generated file directly), the gate fails the local
pre-commit and CI.

### 12.2.2 The manifest

`ai/manifest.json` is the eslint rule's entry point. Shape:

```jsonc
{
  "kerfjsVersion": "0.8.2",
  "files": [
    {
      "name": "skill",
      "source": "kerf.claude-skill.md",
      "bundle": "ai/skill.md",
      "dest": ".claude/skills/kerf-app/SKILL.md",
      "version": "1.0.0",
      "sha256": "…"
    },
    {
      "name": "cursorrules",
      "source": "kerf.cursorrules",
      "bundle": "ai/cursorrules",
      "dest": ".cursorrules",
      "version": "1.0.0",
      "sha256": "…"
    }
  ]
}
```

The manifest is deterministic — `scripts/sync-ai-bundle.mjs` writes the
same bytes given the same inputs, so the in-sync gate can re-run the
generator and compare. There is intentionally no `generatedAt`
timestamp; the git commit time is the actual provenance.

- `kerfjsVersion` is the package's own `version` (read from
  `package.json` at sync time). Informational; **not** the staleness
  signal. Including it does mean a `kerfjs` version bump requires
  re-running the sync, but that's a feature — it forces the
  contributor to confirm `ai/` is current at release time.
- `files[].version` is the per-file semver; bumped by hand when the
  canonical content of that one file changes meaningfully (§12.3.2).
  This IS the staleness signal.
- `files[].sha256` is the hash of the canonical *section* of the
  bundled file (everything above and including the marker). The eslint
  rule uses it as a defensive cross-check against version-bump-without-
  content-change mistakes; it does NOT hash the consumer's local copy.
- `files[].dest` is where the eslint rule installs the file in a
  consumer project, relative to the project root.

## 12.3 The canonical-file contract — version + marker

Each bundled file carries two pieces of structure that the eslint rule
relies on. Together they define what's "kerf's" vs. "the consumer's"
inside one file.

### 12.3.1 The version line

The eslint rule reads a single line of metadata to determine staleness:

```
kerf-skill-version: <semver>
```

Where the line appears depends on the file format:

- `skill.md` carries it inside its YAML frontmatter:

  ```yaml
  ---
  name: kerf-app
  description: …
  kerf-skill-version: 1.0.0
  ---
  ```

- `cursorrules` has no frontmatter, so the line is wrapped in an HTML
  comment as the file's first line:

  ```
  <!-- kerf-skill-version: 1.0.0 -->
  # kerf.cursorrules — …
  ```

The eslint rule's parser is the same in both cases: a regex match on
the first ~512 bytes of the file. It does not need to know whether
YAML or HTML wraps the line.

### 12.3.2 When to bump the version

Bump `files[].version` in the source-of-truth root file (not the
mirror — the sync script propagates) whenever the canonical content
changes in a way a consumer would benefit from re-syncing:

- A hard rule was added, removed, renumbered, or significantly
  rephrased.
- A canonical pattern was added, removed, or materially changed.
- The public API surface in the file's table changed.
- A new common-error row was added that consumers should see.

Do NOT bump for:

- Typo fixes, grammar polish, wording tweaks.
- Comment-only changes that don't alter guidance.
- `kerfjs` version bumps that didn't touch this file's content.

A patch bump is the default; a minor bump signals "consumers should
review their customisations against the new canonical." Major bumps are
reserved for marker-format or version-format breaking changes.

### 12.3.3 The marker

The last line of the canonical content is a one-line marker:

```
<!-- KERF-APP-CANONICAL-END · your customizations below -->
```

Identical text in both `skill.md` and `cursorrules`. The HTML-comment
form means neither Cursor nor Claude Code render it visibly, but the
eslint rule can grep for it unambiguously.

Everything ABOVE the marker (including the marker line itself) is
kerf's canonical section. Everything BELOW is the consumer's append
zone — kerf will never touch it. The sync script writes the bundled
files with an empty append zone (just a trailing newline after the
marker); consumers add their own project-specific guidance there.

## 12.4 The ESLint rule — `kerfjs/ai-assistant-configs`

Implemented as a new rule in
[`eslint-plugin-kerfjs`](../eslint-plugin/README.md). Joins the other
AST rules but is fundamentally different in shape — it doesn't
look at AST nodes; it consults the filesystem.

### 12.4.1 Trigger logic — which projects get nudged

The rule resolves the bundled manifest with
`require.resolve('kerfjs/ai/manifest.json', { paths: [cwd] })`. If the
consumer doesn't have `kerfjs` installed it does nothing (the rule
can't run without a manifest to compare against — it's silent rather
than throwing).

Per-file:

- `skill.md`: triggered when `.claude/` exists in the project root, OR
  when `package.json` has `kerfjs` in `dependencies` /
  `devDependencies` and any `.claude/` directory exists. The rule
  doesn't push Claude Code on consumers who haven't signalled they
  use it (no `.claude/` ⇒ silent).
- `cursorrules`: triggered when `.cursorrules` exists at the project
  root, OR when `.cursor/` exists, OR (same kerfjs-in-deps heuristic).

The "kerfjs-in-deps AND `.claude/` exists" rule catches the realistic
case of a project that uses Claude Code (a `.claude/` directory is
present for other skills) but hasn't installed the kerf-app skill
specifically.

### 12.4.2 Three states per file

For each triggered file, the rule classifies into one of three states:

- **Missing.** The consumer's `dest` path doesn't exist. Report
  recommending `--fix`; auto-fix copies the bundled file in verbatim.
- **Stale.** The file exists, has a valid `kerf-skill-version` line,
  and its version is older than the manifest's. Report recommending
  `--fix`; auto-fix follows the versioned-section semantics below.
- **Up-to-date.** Version matches the manifest. Silent.

A fourth implicit state, **forked**, gets a different report — see
§12.4.4.

### 12.4.3 Auto-fix behaviour — versioned-section preservation

This is the option-2 strategy from KF-217. On `eslint --fix` against
a stale file, the rule:

1. Reads the consumer's local file.
2. Locates the marker line.
3. Reads the bundled file from `node_modules/kerfjs/ai/<bundle>`.
4. Replaces everything **up to and including the marker line** with
   the bundled file's canonical section.
5. Preserves everything **after the marker line** verbatim — the
   consumer's append zone is written back byte-for-byte.
6. Writes the result to the consumer's `dest` path.

A missing file is the degenerate case: the entire bundled file is
written, with the marker at the end and an empty append zone after it.

`eslint --fix` is already a user-initiated invocation, so writing
outside the linted file is consistent with how `--fix` works in
general. The rule's docs explicitly call out the cross-file write —
unusual enough to be worth surfacing.

### 12.4.4 Edge cases — when auto-fix refuses

The contract is "above the marker is kerf's; below is yours." Three
shapes break that contract and the rule refuses to auto-fix, reporting
a "forked" state instead:

- **No marker present** in the consumer's file. Either the file
  pre-dates the marker convention, the consumer deleted the marker,
  or the file is a hand-written variant. Don't guess at the boundary;
  warn and recommend either restoring the marker or disabling the
  rule.
- **Content above the marker has been edited** — the bundled
  canonical's `sha256` (in the manifest) does not match the sha256
  of the consumer's above-marker section. The consumer has chosen to
  customise inside kerf's zone; treat it as a deliberate fork.
- **Multiple markers** in the consumer's file. A well-formed file
  has exactly one. Multiple markers mean the file shape is
  unrecognised.

In all three cases the report names the specific shape mismatch and
points at one of two resolutions: restore the file to the canonical
shape (move customisations below a freshly-inserted marker, delete
extras) or add `'kerfjs/ai-assistant-configs': 'off'` to the eslint
config for that file.

### 12.4.5 Once-per-lint-run

ESLint rules execute per source file, so a naive implementation would
fire the report once per linted file. The rule uses a module-level
guard so the filesystem check runs at most once per lint process; the
report (if any) attaches to whichever source file ESLint happens to
lint first.

The slight UX awkwardness of "the warning shows up on a random
source file" is acceptable: the message is about a project-level
config issue, not about that specific file, and the message text
makes that clear.

### 12.4.6 Disabling

Standard ESLint mechanisms:

```js
// eslint.config.js — turn it off entirely:
{ rules: { 'kerfjs/ai-assistant-configs': 'off' } }

// Or disable only one tool:
{ rules: { 'kerfjs/ai-assistant-configs': ['warn', { claude: false, cursor: true }] } }
```

The granular `{ claude, cursor }` options each default to `true`.

The rule lives in `kerfjs.configs.recommended` at severity `warn`, not
`error` — drift is a nudge, not a hard failure. Consumers who want it
as an error can override per-project.

## 12.5 Design rules

The contract above is rigid on purpose; future contributors editing
the canonical files or the rule should keep these properties intact.

### 12.5.1 Single source of truth at the repo root

`kerf.claude-skill.md` and `kerf.cursorrules` at the repo root are the
only files a human edits. Everything under `ai/` is generated. Never
hand-edit `ai/skill.md`, `ai/cursorrules`, or `ai/manifest.json` — the
in-sync gate will fail.

### 12.5.2 Version bumps are deliberate

Don't auto-bump `kerf-skill-version` on every commit. Bump it when
the change rewards consumers for re-syncing (§12.3.2). The rule's UX
depends on `warn` only firing when there's something the consumer
should see.

### 12.5.3 The marker is structural, not decorative

Don't rephrase or restyle the marker string. The rule's parser is a
strict-text match; any drift in the marker text would silently
classify well-formed files as forked.

### 12.5.4 Zero runtime cost in the kerfjs package

The bundled `ai/` directory is data, not code — it ships as static
files, not as part of the JS bundle, and `import 'kerfjs'` never
loads it. A consumer who installs `kerfjs` but never uses the
eslint plugin pays exactly the ~24 KB of bundled markdown on disk
and nothing at runtime.

### 12.5.5 No postinstall scripts

The bundling-plus-rule strategy was chosen specifically to avoid
postinstall scripts as a discovery mechanism. Postinstall is
increasingly disabled in security-conscious orgs and CI environments;
the eslint rule fires on the normal lint pass that the consumer's
existing CI is already running.

## 12.7 Adding a new bundled file

If a future AI tool needs a similar drop-in (say, `kerf.windsurfrules`
or an `.github/copilot-instructions.md`), the steps are:

1. Create the source-of-truth file at the repo root.
2. Add a `kerf-skill-version: 1.0.0` line in the appropriate
   wrapper (frontmatter or HTML comment) and a
   `<!-- KERF-APP-CANONICAL-END · your customizations below -->`
   marker at the end of its canonical content.
3. Add the file to the `FILES` array in
   `scripts/sync-ai-bundle.mjs` with its `name`, `source`, `bundle`,
   and `dest`.
4. Run `node scripts/sync-ai-bundle.mjs` to regenerate `ai/` (the
   in-sync gate will tell you if you forgot).
5. Add a trigger heuristic to
   `eslint-plugin/lib/rules/ai-assistant-configs.js` —
   "if `.windsurf/` exists, also check this file."
6. Update this doc (§12.6 table + a note in §12.4.1 if the trigger
   logic is non-obvious), the AI usage guide, and the CLAUDE.md
   install section.
7. CHANGELOG entry naming the ticket and the new bundled file.

The contract (`kerf-skill-version` line + marker + bundled in
`ai/` + manifest entry) is rigid on purpose — every new file should
be a copy-paste-and-modify of an existing one, not a fresh design.
